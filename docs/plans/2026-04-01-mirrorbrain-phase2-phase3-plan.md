# MirrorBrain Phase 2 / Phase 3 Plan

## Summary

This plan records the current post-Phase-1 direction for MirrorBrain.

Phase 1 already proved a narrow independent MVP:

- browser memory ingestion through `ActivityWatch`
- local storage and retrieval through `OpenViking`
- a runnable HTTP service and standalone UI
- a basic end-to-end flow across memory review, knowledge draft generation, and skill draft generation

The current goal is not to keep polishing MirrorBrain in isolation. The next major step is to make MirrorBrain usable through the existing `openclaw` plugin system without collapsing MirrorBrain into host-specific code.

Roadmap priority:

1. make MirrorBrain installable and usable through `openclaw`
2. make MirrorBrain genuinely valuable as a personal work memory system
3. evolve toward a personal AI operating system that can help keep user work moving

North-star ordering:

1. users can reliably recover what they previously did
2. users accumulate high-quality knowledge over time
3. users can increasingly rely on AI capabilities shaped by their own work patterns

## Product Direction

MirrorBrain should remain an independent service with an independent UI. That independent surface exists for:

- development and debugging
- operational control and source management
- independent demonstration of the project

But the primary user-facing consumption path should be `openclaw`.

In practice:

- MirrorBrain owns capture, synchronization, memory processing, knowledge generation, and skill generation
- `openclaw` consumes MirrorBrain capabilities
- `openclaw` reuses its own UI and interaction model rather than embedding MirrorBrain's standalone UI as the default user surface

The preferred interface stack is:

- MirrorBrain HTTP service as the canonical independent capability surface
- a thinner `openclaw` plugin-facing wrapper on top of the MirrorBrain service

## Capability Roles

### Memory

Memory is the first integrated capability.

Its immediate user value is:

- a user can ask in `openclaw` what they previously did or saw
- `openclaw` can retrieve relevant past work context from MirrorBrain

Phase 2 memory integration rules:

- retrieval comes before automatic context injection
- user-facing entry starts in `openclaw` chat
- implementation should initially rely on explicit MirrorBrain capability calls under the hood
- the answer shown to the user should be natural language generated by `openclaw`, not a raw retrieval dump
- the answer should include lightweight source hints by default

Default retrieval layering:

- search memory first
- optionally enrich with knowledge
- do not include skills in default chat retrieval

### Knowledge

Knowledge is primarily written for humans and secondarily used by agents.

MirrorBrain, not `openclaw`, owns the workflow that creates knowledge.

`openclaw` should:

- retrieve knowledge when it helps answer a user question
- treat knowledge as a high-quality supplemental source

The next major product focus after Phase 2 is to improve knowledge quality, especially topic- and problem-oriented knowledge artifacts that grow out of daily review.

### Skill

Skill integration follows memory and knowledge integration.

`openclaw` should first consume skills as explicit reusable capabilities. Over time, skills can become more visible to the agent and more capable at execution.

Long-term emphasis:

- prioritize stronger execution capability over merely storing skill assets
- focus first on knowledge-work progression tasks
- optimize first for sustained forward progress, not perfect one-shot output

## Phase Structure

## Current Phase 2 Status Snapshot

The repository now implements the planned Phase 2 baseline in-repo, with explicit Phase 2A integration/demo support and Phase 2B retrieval-quality work for both browser and shell memory.

Implemented so far:

- Phase 2A startup CLI for local developer bring-up
- Phase 2A minimum `query_memory` contract and HTTP exposure
- Phase 2A `openclaw` memory tool example and minimum demo guide
- browser-memory retrieval shaping with:
  - repeated-theme prioritization
  - URL-level representative-source compression
  - normalized browser theme titles
  - more human-readable browser summaries
  - heuristic action summaries for research, documentation, comparison, and debugging
  - combined narrative summaries such as research-plus-documentation and debugging-plus-documentation
  - browser work-recall explanations for `What did I work on yesterday?` style queries
  - browser ranking that prefers action-oriented themes ahead of passive themes when counts are equal
- shell-memory ingestion and sync with:
  - shell-history source support
  - service-level shell sync
  - HTTP and web-UI shell sync entrypoints
- shell retrieval shaping with:
  - command-name grouping for generic shell recall
  - inspection, verification, and apply-phase summaries
  - solve-oriented shell problem-solving sequences
  - shell solve-query explanations
  - solve-oriented shell regrouping even when source types are not explicitly narrowed to shell
- offline narrative generation and storage with:
  - browser theme narrative artifacts for work-recall queries
  - shell problem narrative artifacts with inferred workspace context from command sequences
  - service-level narrative rebuild/publish after explicit browser or shell sync
  - plugin-side preference for stored narratives before raw-event fallback

Remaining non-goal / follow-up territory after Phase 2:

- browser narrative generation is still deterministic and heuristic rather than embedding- or LLM-first
- shell workspace context is inferred from command sequences when explicit cwd/session metadata is unavailable
- Phase 3 knowledge quality work remains the next major product focus

## Phase 2A: Plugin Integration And Minimum Demo

### Goal

Wrap MirrorBrain for `openclaw` plugin use and prove the minimum end-to-end chat retrieval demo.

### Why This Phase Exists

MirrorBrain will not deliver real value if it remains an isolated local demo. The current system must be made easy for `openclaw` developers to install, connect, and exercise in a realistic host workflow.

### Primary Acceptance Criterion

A developer can follow repository documentation, install and configure MirrorBrain for `openclaw`, and run a browser-memory retrieval demo from `openclaw` chat.

### Required Demo Flow

1. start MirrorBrain as an independent local service
2. connect or wrap it as an `openclaw` plugin
3. ask a memory-oriented question in `openclaw` chat
4. have `openclaw` call MirrorBrain retrieval
5. receive a natural-language answer informed by browser memory
6. show lightweight source hints in the answer

### Scope

- developer-installable setup only
- browser memory only for the minimum demo
- HTTP service contract plus `openclaw` plugin wrapper
- integration docs, config docs, and a demo walkthrough

### Non-Goals

- normal-user one-click installation
- shell memory as a prerequisite
- deep knowledge or skill integration before memory retrieval is proven
- moving MirrorBrain's control plane into `openclaw`

### Workstreams

#### 1. Integration Contract

- define the minimum MirrorBrain capability surface needed by `openclaw`
- document request, response, error, and provenance shape for memory retrieval
- keep the wrapper thin and MirrorBrain-owned logic in MirrorBrain

#### 2. Packaging And Setup

- document local service startup
- document plugin wiring into `openclaw`
- define required environment variables and dependency expectations
- make the setup path easy for a developer to reproduce
- provide a dedicated startup script or CLI as the main developer entrypoint
- make that entrypoint start MirrorBrain itself while guiding the user through missing dependencies
- prioritize startup order and dependency readiness before configuration polish
- prioritize configuration clarity before plugin-step completeness

#### 3. Minimum Chat Experience

- use `openclaw` chat as the primary entry
- let `openclaw` compose the final answer from MirrorBrain retrieval results
- include default lightweight attribution in the answer

#### 4. Verification

- add or update integration-level automated coverage where practical
- document the exact manual demo path
- make failure modes visible enough for developers to diagnose setup issues

### Phase 2A Detailed Delivery Notes

#### Startup Script Expectations

The startup entrypoint should be an independent script or CLI rather than documentation-only manual startup.

Its first responsibility is to start MirrorBrain, not just to print preflight checks.

When dependencies are missing, it should:

- stop with clear next-step guidance
- explain which dependency is missing or unhealthy
- explain how to start or validate that dependency

#### OpenViking Setup Support

The first implementation should:

- check service reachability
- inspect whether key configuration is present
- help generate a minimum config template when configuration is missing
- prioritize guidance for embedding or model configuration before lower-value template details
- reuse values from existing `.env` or adjacent local configuration when that is safe and obvious

#### ActivityWatch Setup Support

The first implementation should:

- verify the relevant browser source exists
- verify there are recent browser events in the last hour
- focus on whether browser watcher data is truly flowing, not only whether the service responds

#### MirrorBrain Configuration Support

MirrorBrain should continue using `.env` as the main local configuration surface in this phase.

The first quality bar is:

- missing required variables are reported in one consolidated output
- the report includes example value formats before deeper validation logic is introduced

#### Plugin Example Scope

The first plugin example should cover only `memory retrieval`.

It should document:

- the minimal `openclaw` plugin structure needed
- the required plugin-side configuration
- the smallest possible MirrorBrain call path

#### Minimum Demo Scenario

The first canonical demo question is:

- `我昨天做了什么？`

The preferred answer style is:

- summarize yesterday as 2 to 3 main themes or tasks
- use browser-theme grouping as the primary signal
- use access frequency only as a supporting signal
- include lightweight source hints

The first acceptance path is manual:

- the minimum demo guide should be a step-by-step document
- automated integration coverage is valuable but not the first acceptance gate

#### Phase 2A Execution Backlog

1. add a dedicated startup script or CLI for local developer bring-up
2. make that entrypoint start MirrorBrain and check dependency readiness
3. add strong `OpenViking` diagnostics and minimum-config template generation
4. add `ActivityWatch` browser-data checks for the last hour
5. consolidate MirrorBrain `.env` missing-variable reporting with example formats
6. write the minimum `openclaw` plugin integration example for memory retrieval only
7. write the step-by-step minimum demo guide centered on `我昨天做了什么？`
8. add lightweight manual verification and troubleshooting guidance for the demo path

#### Startup CLI Behavior Contract

The first startup CLI should be:

- guide-like in output style rather than ultra-minimal
- non-interactive
- able to collect and report multiple setup problems in one run
- organized by component in its error summary

Component grouping should include at least:

- MirrorBrain config
- `OpenViking`
- `ActivityWatch`
- MirrorBrain startup

On success, it should:

- start MirrorBrain as a detached child process
- print a complete status summary
- include service address, dependency status, process id, log location, and clear next steps

#### Minimum Memory Retrieval Contract

The first `memory retrieval` contract should stay thin and general-purpose.

Request shape:

- natural-language `query` is the primary input
- optional `time_range`
- optional `source_types`

Default source behavior:

- if no `source_types` are provided, search all enabled sources

Response shape:

- return a list of theme- or task-level memory results
- prefer theme- or task-level units over page-level records

Each memory result should prioritize:

1. `theme` or `title`
2. `summary`
3. `time_range`
4. a small set of representative `source_refs`

Top-level response metadata should prioritize:

- retrieval time-window or interpretation information over raw query echo

#### Minimum OpenClaw Plugin Example

The first example should expose a general tool such as `query_memory`.

Preferred behavior:

- parameters: `query`, optional `time_range`, optional `source_types`
- output: direct pass-through of the MirrorBrain result shape
- invocation: agent chooses to call the tool in response to chat questions
- response composition: `openclaw` summarizes returned results in order
- source hints: each result paragraph gets a lightweight source hint

The example documentation should explain:

- how the tool is registered
- how it calls MirrorBrain
- how the returned results become a final user-facing answer

#### Minimum Demo Guide Structure

The minimum demo guide should optimize first for proving that `openclaw` can call `query_memory` successfully through a realistic chat interaction.

Primary visible success signal:

- the user asks `我昨天做了什么？`
- chat returns a reasonable answer informed by MirrorBrain results

Troubleshooting priority should be:

1. `openclaw` did not invoke `query_memory` correctly
2. MirrorBrain did not return useful memory
3. answer composition produced a poor user-facing response

Guide structure:

- first half: normal step-by-step happy path
- second half: a separate troubleshooting section ordered by the priorities above

## Phase 2B: Memory Retrieval Quality And Experience

### Goal

Make MirrorBrain retrieval useful enough inside `openclaw` that memory recall feels reliable, not gimmicky.

### Priority

Improve MirrorBrain retrieval quality first. Improve host-side chat consumption second.

### Primary Optimization Target

Improve recall accuracy before broader presentation upgrades.

### Data Source Priority

- keep browser memory as the main source to improve first
- expand into shell memory next
- do not prioritize `openclaw` conversation capture during this phase

### Browser Memory Plan

#### Browser Work Order

1. deduplicate and filter noise
2. improve task- or theme-level organization
3. improve time-window understanding

#### Browser Deduplication Rules

- first remove duplicate visit fragments
- first implement URL-level deduplication
- later add semantic deduplication
- use aggressive upper-layer compression for repeated short-window revisits
- preserve lower-layer provenance when needed for traceability

#### Browser Retrieval Shape

- preserve clean single-page memory units at lower layers
- expose theme- or task-oriented grouped memory units as the preferred retrieval layer

#### Browser Grouping Priority

1. page semantic theme
2. search or query chain
3. domain or site context

#### Browser Grouping Implementation Strategy

Phase 2B should start with heuristic grouping rather than heavy semantic clustering:

- page title
- search query
- URL patterns
- time adjacency
- light site context

#### Browser Theme-Memory Refinement

After the Phase 2A minimum demo is working, the first major quality push should start at the upper theme- or task-memory layer rather than only at raw-event cleanup.

The first target question is:

- `昨天/今天我做了什么？`

The desired answer shape is not a list of pages. It should answer what the user worked on, organized by theme.

Priority inside each generated theme memory:

1. identify the theme accurately
2. then describe the main actions or activity around that theme

To improve theme identification:

- prioritize page content summaries or extracted text over titles alone
- strengthen MirrorBrain-side post-processing of already captured browser pages before making the browser-side capture path heavier
- first enrich only high-value pages such as high-frequency pages or pages that repeatedly appear in retrieval
- prioritize theme labels or embeddings before human-readable summaries
- use those semantic outputs first for clustering, not first for ranking

The preferred output of clustering is a theme narrative unit:

- directly describe what the user was doing around that theme
- keep pages as supporting evidence rather than as the primary result

Generation strategy:

- use LLM generation as the primary way to create the theme narrative unit
- generate these narrative units offline during preprocessing
- support both daily generation and later hotspot-based generation, but start with daily generation

The first retrieval-quality outcome should not be "better grouped pages". It should be "better answers about what the user was working on yesterday or today."

#### Browser Theme-Narrative Execution Backlog

1. define the intermediate theme narrative unit for browser memory
2. add MirrorBrain-side enrichment for high-value captured pages
3. generate theme labels or embeddings before human-readable summaries
4. cluster enriched pages into theme groups with semantics first and query-chain signals second
5. generate offline daily theme narrative units with an LLM
6. route retrieval to prefer those theme narrative units over loose page lists
7. evaluate whether answers to `昨天/今天我做了什么？` improve in usefulness and relevance

### Shell Memory Plan

Shell memory matters, but it should not block the browser-memory integration path.

#### Shell Product Goal

Do not build a command log archive. Build task-oriented daily work memory.

#### Shell Signals

- primary source: shell history
- secondary source: logs only as supporting context

#### Shell Representation

- lower layer keeps commands and command fragments
- upper layer should produce daily task-oriented memories

The first high-value shell question is:

- `我之前是怎么通过命令行解决这个问题的？`

The preferred answer shape is a problem-solving narrative rather than a raw command chain.

Inside that narrative, the first priority is:

- recover the key operations the user took
- then clarify the specific problem context around those operations

#### Shell Grouping Rule

Task intent should dominate grouping.

Preferred implementation sequence:

- preserve commands, timestamps, and working directories as facts
- use asynchronous LLM-assisted batch processing to interpret work intent
- generate higher-level task memories after the fact instead of trying to process everything in real time

Signal priority for the first implementation:

- rely first on commands and command sequences
- use logs and output only as secondary supporting context

Preferred abstraction order:

- first recover operation phases instead of only isolated command steps
- for example: identify, verify, apply, re-check

Preferred implementation style:

- use LLM-based interpretation as the primary way to infer phases from command sequences

Generation timing:

- prefer offline asynchronous generation
- support both daily and issue-cluster outputs, but start from issue-cluster generation

Clustering signal order:

1. semantic theme from commands and supporting output
2. workspace or directory
3. time continuity

#### Workspace Log Policy

- logs are secondary enrichment, not the backbone
- only scan inside explicitly authorized workspaces
- first authorization granularity is workspace-level
- first discovery mode should be automatic discovery within the authorized workspace
- only incorporate logs that help identify meaningful work done, not large volumes of noisy intermediate output

### Phase 2B Deliverables

- better browser-memory preprocessing and retrieval quality
- better retrieval-oriented grouping for browser memory
- first shell-memory ingestion and task-memory generation path
- updated docs describing retrieval behavior, source priority, and operator expectations

#### Shell Problem-Narrative Execution Backlog

1. define the intermediate shell problem narrative unit
2. collect shell history as the primary fact source and keep logs as supporting context only
3. group candidate shell activity into issue clusters using semantic signals first
4. infer operation phases from each cluster with LLM assistance
5. generate offline problem-solving narratives centered on key operations taken
6. make shell retrieval answer `我之前是怎么通过命令行解决这个问题的？` before trying to solve broader daily-shell recall

## Phase 3: Knowledge Quality

### Goal

Make MirrorBrain's knowledge artifacts genuinely useful to humans, not just passable outputs for agent retrieval.

### Priority Order

1. better summarization
2. stronger structure and reasoning
3. more ability to inspire future work

### Preferred Knowledge Unit

Topic- or problem-oriented knowledge artifacts, not daily summaries, should become the main durable reading unit.

### Knowledge Growth Model

- daily review remains an important production workflow
- topic knowledge should grow out of repeated daily review input
- the user-facing artifact should remain a rewritten current-best version, not a visible append-only log

### Knowledge History Policy

Keep both:

- lightweight version history
- source provenance back to the reviewed inputs

Priority:

- first keep simple update history and recency visible
- then ensure strong provenance links

### Organizing Principle

Prioritize problem- or theme-oriented knowledge organization before project-only organization.

### Phase 3 Deliverables

- improved topic knowledge synthesis quality
- better workflows for merging daily review output into current-best topic knowledge
- stronger readability for humans
- better metadata for recency and provenance

## Later Skill Roadmap

Skill work remains important, but it should build on stronger memory and knowledge layers.

### Direction

- prioritize stronger execution capability over passive skill curation
- first target knowledge-work progression tasks
- first target clear task goals rather than large long-running projects
- optimize first for sustained progress, not perfect final output quality

### Execution Safety Direction

Potential lower-risk automation may expand over time, but only after explicit policy is documented.

Default safety stance until then:

- information-processing actions are the most plausible automation candidates
- local project operations may become conditional candidates later
- remote or external-system actions still require explicit confirmation by default

## Cross-Cutting Planning Rules

### Installation And Operations

For the current roadmap, installability matters more than theoretical capability breadth.

Before claiming a roadmap item is done, documentation should make it possible for another developer to:

- install the needed local dependencies
- start MirrorBrain independently
- connect it to `openclaw`
- run the intended demo or workflow
- diagnose common failures

### Independent-Service Rule

MirrorBrain must remain independently runnable and debuggable even while optimizing for `openclaw` usage.

### Boundary Rule

Keep capture, synchronization, memory shaping, knowledge creation, and skill generation inside MirrorBrain. Keep `openclaw` integration code focused on adaptation, invocation, and presentation.

### Safety Rule

Even when the roadmap prioritizes usability and execution power, authorization, provenance, and review boundaries remain mandatory constraints rather than optional polish.
